# Cookflow Android — 검증 레짐 (다중 에이전트 · 빡센 검증 루프)

> 빌드(ralph-loop)와 별개로, **각 단계(phase) 완료 시마다 다수 서브에이전트로 적대적·반복 검증**을 돌려 빈틈을 찾고 보정 스토리를 주입한다.
> 목적: "대충 됐음"을 막고, 요구(USER_REQUESTS §7)와 UI_FEEDBACK·PRODUCT_BRIEF 준수를 강제.

## 1. 검증 렌즈 (라운드마다 병렬 에이전트로)
각 에이전트는 읽기 전용으로 코드/스크린샷/문서를 보고 **구체 발견점([심각도]·파일:위치·문제·수정안)** 만 반환.
1. **빌드/아키텍처 정합성** — assembleDebug green, Compose 구조, 파일 200~300줄, 안티패턴, PRODUCT_BRIEF 대비 스코프 드리프트.
2. **다크 디자인 준수** — v3_2 색(#0A0A0A/#161616/#FF6B35/#F0EBE3)·레이아웃·간격 일치, 토큰 사용.
3. **UI_FEEDBACK 준수** — 타이포 14~16sp·행간·한글 줄바꿈, 8dp 그리드, 48dp 터치, 정보밀도, 고정 하단탭바.
4. **모든 버튼 동작 (★)** — dead UI 0: 모든 탭 요소가 화면 이동 또는 최소 안내(alert), press 피드백.
5. **핸즈프리 UX** — step-gated·페이싱(자동진행 금지)·플로팅 비서·showVideo 슬라이드·타이머·음성명령 UI.
6. **영상 연동 정확성 (PB/PD)** — seek/볼륨 더킹/loop/단계매칭이 **실제로** 동작하는 증거(스크린샷/로그) 적대적 확인. 더킹 가능/대안 결론.
7. **접근성** — 대비 AA, contentDescription, 터치 타깃, 동적 상태 안내.
8. **한국어 카피/페르소나 톤** — 영어 누수, 이모/엄마/셰프 톤 차이.

## 2. 단계 게이트 (이 시점마다 검증 라운드 실행)
- **PA 후**: 렌즈 1,2,3 (디자인 시스템·토큰·구조)
- **PB(스파이크) 후**: 렌즈 6 집중 + 1 — **영상 4종 실증을 적대적으로** (정말 됐나? 증거?)
- **PC(5화면) 후**: 렌즈 2,3,4,8 (화면 디자인·모든 버튼·카피)
- **PD(핸즈프리) 후**: 렌즈 5,6,7 (UX·영상·접근성)
- **PE/완주 후**: 렌즈 1~8 전체 + 종합

## 3. 적대적 검증 (findings 신뢰도)
- 각 발견점은 **회의론자(skeptic) 에이전트**가 반박 시도 — "이거 진짜 문제 맞나? 증거?". 반박되면 폐기.
- 영상 동작 같은 핵심 주장은 **스크린샷/로그 증거 없으면 미검증으로 처리**(주장만으로 통과 금지).

## 4. 보정 주입 & 반복 (loop-until-clean)
- 확정된 발견점 → 보정 스토리(UI-Rn/FIX-n)로 prd.json에 idempotent 주입(경합 회피: codex 미실행 시) → ralph 루프 재가동.
- **2라운드 연속 새 발견 0** 이 될 때까지 단계 검증 반복.
- 각 라운드 결과는 docs/progress/FANOUT_ROUND_n_ko.md + REPORT_ko.md에 한글 기록.

## 5. 강도 설정
- 라운드당 병렬 에이전트: 해당 단계 렌즈 수(3~8) + 핵심 발견점당 skeptic 1.
- 토큰/시간 제약보다 **정확성·요구 준수 우선**. 자율 진행, 승인 없음.
