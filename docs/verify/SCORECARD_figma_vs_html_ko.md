# Figma 이식본 ↔ app.html 동일성 점수표 (Scorecard)

> 목적: **원본 충실 이식 트랙 `cook/<view>`** 가 `app.html`의 화면·요소·라벨·플로우와 **동일한지**를 객관적으로 채점하고, **총점 95점**을 넘길 때까지 보정 루프를 돌린다.
> 대상 아님: 디자인 개선판 `cook-v2/*`(의도적으로 원본과 다름 — 별도 트랙).

## 채점 원칙 (반드시 준수)
1. **사실 기반 채점**: 점수는 에이전트의 주관 판단이 아니라, Figma `scan_text_nodes`/`get_node_info`가 **실제로 반환한 텍스트·노드 존재/부재**로 매긴다. 각 기준은 `있음/없음`으로 객관 판정.
2. **증거 필수**: 채점 리포트는 화면별로 `기대 N개 / 발견 M개 / 누락 [목록]`을 반드시 남긴다. 근거 없는 "92점"은 무효.
3. **읽기 전용·적대적 채점기**: 채점 에이전트는 Figma에 **쓰지 않는다**(불일치를 찾는 역할). 보정은 **기존 포팅 루프(단일 writer)** 가 수행 → 트윈 충돌 방지.
4. **와이어프레임 의도 반영**: 픽셀 동일성이 아니라 **구조·라벨·플로우 동일성**. 아이콘은 placeholder 허용, 폰트 렌더 차이 무시.
5. **Ground truth**: `docs/verify/CHECKLIST_app_html_ko.md`(app.html에서 추출한 화면별 기대 요소·exact 한글 라벨·버튼·플로우) + `scripts/ralph-figma/prd.json`의 `acceptanceCriteria`.

## 대상 화면 (11)
home · detail · cook3 · cook · cook2 · loading · complete · sheet · tipWrite · cookbook · reviews

## 화면별 채점 (각 화면 100점)
| 차원 | 배점 | 측정 방법 (객관) |
|---|---:|---|
| A. 요소 완전성 | 35 | 체크리스트의 핵심 구조 섹션(상태바/헤더/콘텐츠 블록/하단탭바 등)이 `cook/<view>` 프레임에 존재. 점수 = (발견 요소/기대 요소)×35. `get_node_info`로 자식 프레임·요소 확인. |
| B. 라벨 일치 | 30 | 체크리스트의 **exact 한글 라벨**이 `scan_text_nodes` 결과에 존재. 점수 = (매칭 라벨/기대 라벨)×30. 동적(JS 생성) 라벨은 대표 1개로 대체 인정. |
| C. 레이아웃 순서 | 10 | top→bottom 섹션 순서가 app.html과 일치(요소 y좌표 순). 어긋난 섹션 1개당 −2. |
| D. 팔레트/모드 | 10 | 라이트(크림 #F6F0E8)/다크(cook·cook2·cook3 #050505·#120F0C) 모드 정확, 주색 테라코타 #D66B42, **초록 오용 없음**(진짜 초록 #3FA76F은 성공만). 위반 1건당 −3. |
| E. 정직성·dead UI 0 | 5 | cook3 영상 거짓 "재생 중" 없음/정직 폴백 표기, 모든 버튼이 의미 있는 액션. 위반 1건당 −2. |
| F. 플로우 연결 | 10 | 이 화면의 진입·진출 연결(`create_connections` 화살표)이 플로우 그래프와 일치. 누락 연결 1개당 −3. |

## 총점 산출
- **화면 점수** = A+B+C+D+E+F (0~100).
- **총점** = 11개 화면의 **가중 평균**. 가중치: 주력 화면 `home·detail·cook3` = **1.5**, 그 외 = **1.0**.
- **통과 조건(둘 다 충족)**:
  1. 총점 **≥ 95**
  2. **모든 화면 ≥ 90** (한 화면이 다른 화면을 가려 통과하는 것 방지)

## 채점 + 보정 루프 (95점까지)
```
전제 게이트: 포팅 113/113 완료 AND 플러그인 라이브(같은 채널). 미충족이면 채점 중단.
반복(최대 5라운드):
  1) 채점(읽기전용): 각 cook/<view>에 join→scan_text_nodes+get_node_info로
     체크리스트 대비 발견/누락 판정 → 화면점수·총점 계산 →
     docs/verify/SCORE_REPORT_ko.md 에 증거(발견/누락 목록)와 함께 기록.
  2) 통과 조건 충족 → 종료(<promise>SCORE_PASS</promise>).
  3) 미충족 → 누락 항목마다 prd.json 보정 스토리 생성(passes=false):
     acceptanceCriteria = "cook/<view>에 <누락 라벨/요소> 추가·수정".
  4) 기존 포팅 루프(단일 writer)로 보정 스토리 그리기.
  5) 재채점(1로).
라운드 상한 도달 시: 남은 갭을 리포트에 명시하고 사람 확인 요청(무한루프 금지).
```
